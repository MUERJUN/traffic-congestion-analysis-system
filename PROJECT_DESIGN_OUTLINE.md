# 基于数据挖掘的城市道路拥堵预测与交通优化分析系统

## 《项目设计大纲》V1.0（已确认基线）

> 当前状态：已于 2026-07-11 获用户确认；第二阶段仅进行数据获取、数据审计和 EDA 准备，不训练模型。  
> 基线规则：本大纲经确认后，后续需求、数据处理、建模、页面和验收均以此为准；如需调整，先更新大纲并记录变更原因。

## 0. 关键口径与范围边界

### 0.1 研究对象

- 以 METR-LA 中的道路传感器作为道路观测点，以“传感器编号 + 观测时刻”为最小分析单元。
- 系统分析的是洛杉矶县高速公路传感器记录的道路速度状态，不把传感器直接等同于完整城市道路路段。
- 第一版只使用 METR-LA 及其同源传感器元数据，不随意融合其他城市、其他年份或其他采样口径的数据。

### 0.2 核心任务

- 描述性任务：发现时间规律、易拥堵传感器、速度趋势和拥堵前兆。
- 预测任务：使用预测时点之前的历史信息，判断某传感器未来 30 分钟是否会出现“持续拥堵”。
- 解释任务：给出全局影响因素排名，并解释单次预测为何形成高风险判断。
- 决策支持任务：将统计证据与模型解释转化为分级、可执行且不过度承诺的交通优化建议。

### 0.3 拥堵与风险定义（拟定，待数据验证）

- 训练集内按传感器估计自由流速度，默认取该传感器训练期速度的第 85 百分位数，避免不同道路直接使用同一绝对阈值。
- 速度比 = 当前速度 / 该传感器自由流速度。
- 单时点拥堵：速度比低于 0.60。该阈值将在 0.50、0.60、0.70 三档进行敏感性分析后确认。
- 未来 30 分钟持续拥堵标签：未来 6 个五分钟观测中至少 3 个被判为拥堵，标签为 1；否则为 0。
- 拥堵概率：分类模型输出的正类概率，必要时使用验证集进行概率校准。
- 风险等级：低风险 `[0, 0.40)`、中风险 `[0.40, 0.70)`、高风险 `[0.70, 1.00]`。阈值最终结合验证集召回率、误报率与展示需求确定。
- 所有自由流速度、缺失值填补、标准化和阈值统计只由训练集计算，严禁利用未来数据。

### 0.4 非目标

- 不做真实信号灯、匝道或导航系统的自动控制。
- 不宣称模型输出能够替代交通管理人员决策。
- 不在第一版引入深度学习、实时流处理、微服务、大数据集群或复杂地图引擎。
- 不把历史数据 Demo 描述成真实生产环境的实时系统。

## 1. 项目定位

### 1.1 项目名称

基于数据挖掘的城市道路拥堵预测与交通优化分析系统。

### 1.2 项目性质

面向课程展示的、可复现的道路交通数据挖掘与辅助决策 Demo。项目重点是完成“问题—数据—分析—模型—解释—建议—展示”的完整闭环，而不是单纯追求模型精度。

### 1.3 核心价值

- 从事后发现转向未来 30 分钟风险预警。
- 从单一速度展示转向道路、时间和趋势的多维规律分析。
- 从黑盒概率转向“概率 + 主要原因 + 历史依据”的解释输出。
- 从分析结论转向与风险等级对应的管理建议。

### 1.4 成功标准

- 数据来源、字段、时间范围和清洗过程可追溯。
- EDA 的每张关键图表均对应一条有证据的业务结论。
- 至少比较 Logistic Regression、Random Forest、XGBoost 三类模型。
- 测试集输出 Accuracy、Precision、Recall、F1-score、ROC-AUC，并说明类别分布和混淆矩阵。
- 任一高风险预测能展示至少三个主要影响因素。
- Demo 三类页面可完整串联数据分析、风险预测和分析报告。
- 固定数据、随机种子和模型版本下结果可复现。

## 2. 行业痛点分析

### 2.1 事后响应

传统监控通常在速度已经明显下降后才告警，管理人员缺少提前量，无法预先开展诱导、分流或重点监测。

### 2.2 数据利用不足

传感器持续产生高频速度数据，但若只展示当前数值，会浪费其中的周期规律、趋势变化和历史同期信息。

### 2.3 风险道路识别效率低

多传感器同时运行时，人工难以快速比较各道路未来风险并确定优先处理对象。

### 2.4 原因不可解释

只有“拥堵/不拥堵”结论无法回答当前速度是否持续下降、历史同期是否高发、是否正处于通勤高峰等管理问题。

### 2.5 建议缺少证据链

笼统的“加强管理”缺乏行动对象、触发条件和数据依据。项目需建立“现象—因素—风险—建议”的证据链。

### 2.6 数据现实与管理需求之间的差距

METR-LA 主要提供速度数据，缺少事故、施工、车流量、车道占用率、天气及控制措施结果，因此能可靠解释的是统计关联和速度变化模式，不能直接证明事故等具体因果，也不能量化优化措施实施后的真实收益。

## 3. 用户需求分析

### 3.1 目标用户

| 用户 | 核心问题 | 系统输出 |
|---|---|---|
| 交通管理人员 | 哪些观测点未来可能拥堵，先关注哪里 | 风险排行、概率、等级、关键原因 |
| 交通分析人员 | 何时、何地容易拥堵，变化规律是什么 | 时段规律、道路排行、趋势、拥堵前兆 |
| 项目评审/教师 | 数据挖掘流程是否完整、模型是否可信 | 数据说明、实验对比、解释、局限、复现说明 |
| 系统演示者 | 如何快速完成一次完整业务演示 | 预设样例、交互筛选、预测卡片、报告摘要 |

### 3.2 功能需求

- 查看数据规模、时间范围、传感器数量、缺失率及标签分布。
- 按日期类型、小时、传感器和拥堵状态筛选分析结果。
- 查看小时规律、工作日/周末差异、易拥堵传感器排行、速度趋势和拥堵前窗口变化。
- 输入传感器编号并选择一个数据集中可用的历史时点，输出当前状态、未来 30 分钟拥堵概率和风险等级。
- 展示单次预测的主要原因及对应特征值。
- 查看三类模型的统一测试结果、混淆矩阵、ROC 曲线、影响因素排名和优化建议。

### 3.3 非功能需求

- 易用性：主要演示流程不超过三次页面切换。
- 性能：模型和汇总结果预计算；普通电脑上单次页面交互目标在 3 秒内完成。
- 可复现性：固定时间切分、随机种子、依赖版本和模型参数。
- 可维护性：数据、特征、模型、报告、页面逻辑分层。
- 可信性：清楚区分观测事实、统计关联、模型预测和管理建议。
- 隐私性：数据不含个人身份信息；不采集用户隐私数据。

### 3.4 典型用户故事

- 作为管理人员，我选择传感器和历史时点后，希望看到未来 30 分钟风险及前三个原因，以决定是否重点监测。
- 作为分析人员，我希望比较各时段与各传感器的拥堵率，以识别周期性高风险对象。
- 作为评审，我希望看到严格的时间切分、模型对比和解释方法，以判断结论是否存在泄漏或夸大。

## 4. 数据来源设计

### 4.1 核心数据集

- 数据集：METR-LA Traffic Dataset。
- 推荐获取渠道：DCRNN 作者公开仓库所指向的数据文件。
- 主要文件：速度矩阵 `metr-la.h5`；同源传感器编号、位置、道路距离或邻接信息作为可选元数据。
- 已知结构：约 207 个道路传感器，五分钟采样，核心值为平均速度；常见版本覆盖 2012-03-01 至 2012-06-30。实际范围、行列数和单位以下载后的数据审计结果为准。
- 基础表转换：将“时间索引 × 传感器列”的宽表转换为“timestamp、sensor_id、speed”的长表，便于分析与特征构造。

### 4.2 字段规划

| 字段 | 类型 | 来源/生成方式 | 用途 |
|---|---|---|---|
| timestamp | 时间 | 原始索引 | 排序、切分、时间特征 |
| sensor_id | 类别 | 原始列名 | 道路观测点标识 |
| speed | 数值 | 原始观测 | 核心状态变量 |
| latitude/longitude | 数值，可选 | 同源传感器元数据 | 传感器位置展示 |
| hour/day_of_week/is_weekend | 派生 | timestamp | 周期规律与建模 |
| lag/rolling/trend features | 派生 | 历史 speed | 预测与前兆分析 |
| free_flow_speed/speed_ratio | 派生 | 训练集统计 | 跨传感器相对状态 |
| congestion_label | 派生 | 未来 30 分钟规则 | 分类目标 |

### 4.3 数据版本与追溯

- 保存原始文件，不直接修改。
- 记录下载来源、下载日期、文件大小、哈希值、字段结构与许可/引用信息。
- 分设 raw、interim、processed、metadata 目录；分析和训练只读取受控版本。
- 生成数据字典与数据质量报告。

### 4.4 辅助数据决策

第一版不增加天气数据，原因如下：

1. 当前课程目标可由速度历史、时间周期和同源元数据完成；
2. METR-LA 的传感器位置、2012 年历史天气的空间站点和时区需要额外校准；
3. 不同来源可能存在时间粒度、缺失机制和地点匹配误差；
4. 在没有先建立单数据源基线前，引入天气会降低问题可解释性和实验可控性。

若后续经确认增加天气，必须单独提出数据变更：按统一时区将天气重采样为五分钟或小时粒度，通过最近气象站/空间距离关联传感器，并报告匹配距离、覆盖率、时间误差、缺失率和一致性风险；新增模型必须与无天气基线做消融对比。

### 4.5 数据限制

- 数据年代较早且覆盖期约四个月，无法代表所有年份和季节。
- 传感器覆盖高速公路网络，不代表普通城市支路和交叉口。
- 速度为传感器观测，可能存在零值、异常值、连续缺失和设备故障。
- 仅凭速度与时间变量不能建立交通事件的因果解释。

## 5. 数据处理流程

### 5.1 数据审计

- 校验文件可读性、维度、时间范围、采样间隔、时区信息、传感器数量与重复编号。
- 统计总记录数、每个传感器覆盖率、缺失率、零值率、速度分布和连续缺失长度。
- 检查时间戳重复、逆序、断点及非五分钟间隔。
- 输出数据审计表，未经审计不进入建模。

### 5.2 清洗原则

- 去重并按 sensor_id、timestamp 排序。
- 负速度和明显非法速度标记为缺失，不直接替换为正常值。
- 零值先结合持续长度和相邻时刻判断；不能一律视为真实拥堵，也不能一律删除。
- 短缺失段可使用只依赖过去信息的前向填充或训练期同传感器历史同期中位数；长缺失段保留质量标记并排除相应训练样本。
- 所有填补规则、阈值和删除数量写入质量报告。

### 5.3 时间与派生特征

- 时间特征：小时、星期、是否周末、五分钟时段编号；周期变量使用正余弦编码供模型使用。
- 历史滞后：5、10、15、30、60 分钟速度或速度比。
- 滚动统计：过去 15、30、60 分钟的均值、最小值、标准差。
- 趋势特征：最近 10/15/30 分钟速度斜率、相邻差分、连续下降次数。
- 历史同期：仅用训练期统计得到的 sensor_id × 星期类型 × 时段的历史中位速度与历史拥堵率。
- 传感器特征：sensor_id 作为类别信息；优先使用相对速度与历史统计，避免把编号误解为连续数值。
- 可选空间特征：同源邻接元数据可用于构造邻近传感器过去时刻平均速度，但必须证明时间对齐且不得使用未来值；第一版可作为扩展实验，不作为最低交付依赖。

### 5.4 标签构造

- 对每个传感器按时间生成未来六个观测点的状态。
- 使用第 0.3 节定义生成二分类标签。
- 数据尾部没有完整未来窗口的样本不用于监督训练。
- 特征截止时间必须早于标签窗口起点，形成自动化防泄漏检查清单。

### 5.5 数据划分

- 按全局时间顺序划分训练集 70%、验证集 10%、测试集 20%，不随机打乱。
- 边界处保留必要间隔，防止滚动窗口跨集合共享未来信息。
- 训练集用于拟合预处理与模型；验证集用于调参与阈值选择；测试集仅用于最终一次评价。
- 如需更稳健评估，在训练/验证阶段采用时间序列滚动验证，不用普通随机 K 折。

### 5.6 类别不平衡处理

- 先报告拥堵与非拥堵比例。
- Logistic Regression 和 Random Forest 优先使用类别权重；XGBoost 使用正负样本比例权重。
- 不默认采用随机过采样；若实验使用，只能在训练折内部进行并记录影响。
- 选择阈值时重点权衡拥堵类 Recall、Precision 和 F1，不只追求 Accuracy。

### 5.7 处理产物

- 清洗后的长表。
- 特征表和字段字典。
- 时间切分清单。
- 数据质量报告、标签分布报告和泄漏检查结果。

## 6. 数据分析方案

### 6.1 分析总原则

每个分析主题按“问题—指标—图表—发现—业务含义—限制”六项输出。图表不得只陈列现象，必须形成可复核的业务结论。

### 6.2 不同时间段拥堵规律

- 问题：一天和一周中哪些时段速度较低、拥堵率较高？
- 指标：平均/中位速度、速度分位数、拥堵率、样本量。
- 分组：小时、五分钟时段、星期、工作日/周末。
- 图表：24 小时速度曲线、小时×星期热力图、工作日与周末对比。
- 预期结论形式：“工作日 7—9 时速度中位数较非高峰下降 X%，拥堵率提高 Y 个百分点，说明通勤时段存在稳定周期性压力。”实际 X、Y 必须由数据计算。

### 6.3 易拥堵道路/传感器识别

- 问题：哪些传感器拥堵频率高、持续时间长且波动明显？
- 指标：拥堵率、拥堵事件次数、平均/最长持续时间、低速分位数、有效样本量。
- 图表：Top N 排行条形图、传感器风险矩阵、可选位置散点图。
- 排名规则：设最小有效样本门槛，避免缺失严重的传感器进入高风险榜。
- 结论区分“经常拥堵”和“偶发但严重拥堵”。

### 6.4 道路速度变化趋势

- 问题：整体及单传感器速度如何随时间变化？
- 指标：五分钟速度、小时/日聚合速度、滚动均值、滚动波动率。
- 图表：时间序列折线、日趋势、分位带、选定传感器对比。
- 结论需区分长期趋势、周期变化与短时异常，不把平滑曲线当作原始事实。

### 6.5 拥堵发生前的数据变化特点

- 问题：拥堵前 60 分钟是否出现速度下降、波动加剧或连续下降？
- 方法：以拥堵事件首次发生点为 t=0，对齐前 60 分钟窗口；与相同时段的非拥堵对照窗口比较。
- 指标：相对速度、速度斜率、标准差、连续下降次数、与历史同期偏差。
- 图表：事件前相对时间曲线、拥堵/非拥堵箱线图、特征分布对比。
- 统计验证：报告效应大小和置信区间；必要时使用非参数检验，避免只凭视觉判断。
- 结论示例：“高风险事件前 15 分钟的速度下降斜率显著大于对照窗口，说明持续下降可作为短时预警信号。”

### 6.6 数据质量与异常专题

- 展示缺失率、零值率和传感器覆盖差异。
- 分析零值处理前后关键结论是否变化。
- 对拥堵阈值开展敏感性分析，确认道路排行与时段规律是否稳定。

### 6.7 EDA 最终业务结论模板

每条结论包含：对象、时间范围、定量证据、可能解释、管理含义、限制。例如：

> 传感器 S 在工作日 07:00—09:00 的拥堵率为 A%，较其非高峰高 B 个百分点；其拥堵前 15 分钟中位速度下降 C%。因此该点位应列为早高峰预警重点，但该结论反映统计关联，不能仅凭现有数据断定具体事故或流量原因。

## 7. 模型方案

### 7.1 问题形式

- 主任务：未来 30 分钟持续拥堵二分类。
- 输入：预测时点及之前的时间、历史速度、滚动统计、趋势和历史同期特征。
- 输出：拥堵类别、拥堵概率、风险等级。
- 建模粒度：sensor_id × timestamp；统一模型覆盖全部传感器，必要时通过传感器类别和相对特征体现差异。

### 7.2 基线与候选模型

| 模型 | 定位 | 主要价值 |
|---|---|---|
| Dummy/历史同期规则 | 最低基线，不参与最终三模型要求 | 判断机器学习是否真正超过简单策略 |
| Logistic Regression | 线性、透明基线 | 系数方向清晰，便于解释 |
| Random Forest | 非线性集成模型 | 捕捉特征交互，对缩放不敏感 |
| XGBoost | 强化树模型 | 对表格特征通常有效，支持局部贡献解释 |

不在第一阶段加入 LSTM、DCRNN、Transformer 等深度模型；若传统模型无法达到基本效果，只作为后续独立扩展，不替代本大纲要求的三模型比较。

### 7.3 训练与调参

- 所有模型使用相同的时间划分、标签口径和核心特征集合。
- Logistic Regression 调整正则化强度与类别权重。
- Random Forest 调整树数量、最大深度、叶节点最小样本数和特征采样比例。
- XGBoost 调整学习率、树深、树数量、子采样、列采样和类别权重，并使用验证集早停。
- 控制搜索空间，优先使用小规模随机搜索或手工分层实验，避免为了调参扩大项目复杂度。
- 记录参数、随机种子、训练时间和模型文件版本。

### 7.4 评价指标

测试集必须输出：

- Accuracy：总体判断正确比例，但在类别不平衡时不单独使用。
- Precision：预测拥堵中真实拥堵的比例，反映误报成本。
- Recall：真实拥堵被提前识别的比例，反映漏报风险。
- F1-score：Precision 与 Recall 的综合。
- ROC-AUC：不同阈值下的整体排序能力。

同时展示混淆矩阵、ROC 曲线、类别分布；建议补充 PR-AUC 与概率校准曲线作为不平衡和概率可信度的辅助指标。

### 7.5 最终模型选择规则

- 先排除明显过拟合、时间外推不稳定或概率严重失真的模型。
- 以验证集拥堵类 F1 为主指标，Recall 和 ROC-AUC 为重要辅助指标；在 F1 接近时优先 Recall 较高且解释更稳定的模型。
- 最终模型只在选择和阈值固定后于测试集做一次正式评价。
- 不允许因为测试集结果不理想而反复回到测试集调参。
- 最终报告说明为什么选择该模型，以及未选模型的不足。

### 7.6 稳健性与误差分析

- 比较工作日/周末、高峰/非高峰和不同传感器组的效果。
- 检查误报与漏报样本，分析是否与缺失、突发速度跳变、阈值边界有关。
- 检查不同拥堵阈值下模型排名是否稳定。
- 若部分传感器效果很差，页面需给出数据质量或低置信度提示。

### 7.7 模型解释

- 全局解释：Logistic 系数、树模型排列重要性及 XGBoost 的 SHAP 汇总，输出稳定的影响因素排名。
- 局部解释：对单次预测展示前三至五个推动风险上升/下降的因素、实际特征值及通俗解释。
- 方向表达：不能只显示“重要性 0.18”，还要说明“最近 15 分钟速度持续下降，提高了拥堵风险”。
- 稳定性：在不同时间窗口或验证折比较重要性排名，避免把偶然排序当作业务规律。
- 因果边界：解释的是模型依据和统计关联，不表述为已经证明的因果关系。

### 7.8 单次预测解释模板

> 传感器 S 在时刻 T 的未来 30 分钟拥堵概率为 P%，风险等级为高。主要依据：① 当前速度比为 A，低于历史正常水平；② 最近 15 分钟速度下降 B；③ 当前时段历史拥堵率为 C%。该结果用于辅助预警，不代表真实事故诊断。

## 8. 系统架构设计

### 8.1 技术选型

- 数据处理：Python、Pandas、NumPy。
- 机器学习：Scikit-learn、XGBoost。
- 分析绘图：Matplotlib/Seaborn 用于静态报告，Plotly 用于交互页面。
- Web Demo：Streamlit。
- 模型解释：Scikit-learn 排列重要性；最终模型为树模型时使用 SHAP 或等价的树贡献方法。
- 持久化：Parquet/CSV 保存整理数据，Joblib/模型原生格式保存模型，JSON 保存指标与配置。

选择 Streamlit 的原因：课程项目以单机演示、数据交互和快速复现为主，不需要 Flask 的前后端拆分、用户系统和复杂 API。

### 8.2 逻辑分层

1. 数据源层：METR-LA 原始速度和同源元数据。
2. 数据处理层：审计、清洗、长表转换、标签和特征生成。
3. 分析建模层：EDA、训练、评估、解释、建议规则映射。
4. 产物层：处理数据、模型、指标、图表数据、报告。
5. 展示层：数据分析页、预测页、分析报告页。

### 8.3 运行方式

- 离线流程：下载/校验数据 → 清洗 → 生成特征 → 训练比较 → 固化模型与指标。
- 在线演示流程：页面读取预计算汇总与模型 → 用户选择传感器/历史时点 → 构造该时点已有特征 → 输出概率、等级和解释。
- Demo 中的“当前状态”指用户所选历史时点的数据状态，不冒充实时采集。

### 8.4 计划目录（确认后创建，当前不创建代码结构）

- `data/raw`、`data/interim`、`data/processed`、`data/metadata`
- `notebooks`：探索与实验记录
- `src/data`、`src/features`、`src/models`、`src/analysis`、`src/app`
- `artifacts/models`、`artifacts/metrics`、`artifacts/figures`
- `reports`、`tests`、`config`

### 8.5 质量保障

- 数据检查：维度、采样间隔、标签边界、缺失处理数量。
- 特征检查：未来值泄漏、滚动窗口方向、训练统计复用。
- 模型检查：输入字段一致、概率范围、固定样例结果稳定。
- 页面检查：空筛选、无效传感器、缺失时点、模型文件缺失时给出明确提示。

## 9. 页面设计

### 9.1 全局布局

- 顶部：项目名称、数据时间范围、Demo 非实时提示。
- 侧边栏：页面导航、日期类型、时间范围和传感器筛选。
- 统一颜色：畅通为绿色、中风险为橙色、高风险为红色；图例和阈值保持一致。
- 图表下方固定放置“业务解读”与“数据限制”，避免图表与结论分离。

### 9.2 数据分析页面

**目标：** 回答“数据是什么、拥堵何时发生、哪些点位高发、发生前有什么变化”。

**模块：**

1. 数据概览卡片：时间范围、传感器数量、记录数、总体缺失率、总体拥堵率。
2. 数据质量：缺失率分布、零值率、异常传感器提示。
3. 时间规律：小时速度曲线、小时×星期拥堵热力图、工作日/周末对比。
4. 道路排行：易拥堵传感器 Top N，可按拥堵率/持续时间切换。
5. 速度趋势：选择传感器后查看原始速度与滚动趋势。
6. 拥堵前兆：拥堵事件前 60 分钟与对照窗口对比。
7. 业务结论：随筛选联动的定量文字摘要。

### 9.3 预测页面

**输入：**

- 传感器编号（下拉搜索，避免非法编号）。
- 数据集中可预测的历史时点；默认提供一个有代表性的演示时点。

**输出：**

- 当前速度、相对自由流速度、当前状态。
- 未来 30 分钟拥堵概率与低/中/高风险等级。
- 近期 60 分钟速度趋势。
- 主要影响因素 Top 3—5：因素名、特征值、推动方向、通俗解释。
- 同时段历史拥堵率及必要的数据质量提示。
- 分级建议与“历史回放、非实时控制”声明。

**异常状态：**

- 历史窗口不足、关键数据缺失或传感器质量差时，不强行预测，明确说明原因。

### 9.4 分析报告页面

**模块：**

1. 数据与任务定义：数据范围、标签、时间切分。
2. 模型评价表：三模型五项必选指标与最佳模型标记。
3. 评价图：混淆矩阵、ROC 曲线、可选 PR/校准曲线。
4. 影响因素：全局重要性排名、方向和解释。
5. 误差分析：高峰/非高峰、工作日/周末、传感器组差异。
6. 优化建议：按发现类型和风险等级组织。
7. 局限与伦理：数据年代、覆盖范围、因果限制、非实时说明。

### 9.5 建议生成规则

- 高风险且速度持续下降：建议提前监测、发布出行提示、评估分流条件。
- 历史同期持续高发：建议纳入固定高峰管理清单，优化值守与信息发布时段。
- 邻近点位同时恶化（若启用空间特征）：建议关注走廊级传播风险，而非只处理单点。
- 高波动但证据不足：建议复核传感器质量和外部事件，不直接给出控制结论。
- 所有建议标明依据、适用条件和“需管理人员复核”，不输出未经数据验证的精确控制参数。

- 完成功能测试、边界测试、性能检查和视觉一致性修正。

## 11. 风险分析

| 风险 | 影响 | 应对措施 |
|---|---|---|
| 数据下载或格式版本不一致 | 无法复现或字段错位 | 固定官方/作者来源、哈希、数据审计与版本登记 |
| 零值既可能是拥堵也可能是故障 | 标签和结论失真 | 分析连续零值与覆盖情况，设置质量标记并做敏感性分析 |
| 固定绝对速度阈值不适合所有道路 | 拥堵标签偏差 | 使用传感器相对自由流速度并验证多阈值稳定性 |
| 时间序列泄漏 | 测试指标虚高 | 严格时间切分，所有统计仅拟合训练集，增加特征截止检查 |
| 类别不平衡 | Accuracy 虚高、拥堵漏报 | 类别权重、F1/Recall/PR-AUC 辅助分析、阈值优化 |
| 模型过拟合某些传感器或月份 | 泛化能力弱 | 时间滚动验证、分组评估、控制模型复杂度 |
| 只用速度与时间，原因信息不足 | 解释被误认为因果 | 明确“模型依据/统计关联”，不虚构事故、天气或流量原因 |
| 训练数据较旧、覆盖期短 | 现实代表性有限 | 将系统定位为课程研究 Demo，报告外部有效性限制 |
| 传感器与道路概念混淆 | 页面表述不准确 | 统一称“传感器/道路观测点”，有元数据时再展示道路位置 |
| 用户误认为实时预测 | 演示误导 | 使用历史回放时点，页面固定显示非实时声明 |
| XGBoost 提升有限 | 增加依赖但收益低 | 保留 Logistic/RF，严格按验证指标和解释性选择最终模型 |
| SHAP 等解释过度解读 | 把相关性当因果性 | 同时展示特征值、方向、稳定性和因果边界 |
| 页面交互缓慢 | 影响课程演示 | 预计算 EDA 汇总、缓存数据和模型，限制大图默认数据量 |
| 管理建议无法验证效果 | 结论过度承诺 | 建议采用条件式表达，定位为辅助决策，不估算未经实验的收益 |

## 13. 设计基线待确认项

本版本建议一次性确认以下核心决策：

1. 预测窗口固定为未来 30 分钟。
2. 拥堵采用“传感器自由流速度的 60%”相对阈值，并在 EDA 阶段做 50%/60%/70% 敏感性分析。
3. 未来六个观测中至少三个拥堵定义为“未来 30 分钟持续拥堵”。
4. 数据按时间 70%/10%/20% 切分为训练/验证/测试。
5. 第一版不加入天气，不上深度学习。
6. 使用 Streamlit 构建单机历史回放 Demo。
7. 最终模型以验证集拥堵类 F1 为主、Recall 与 ROC-AUC 为辅选择。
