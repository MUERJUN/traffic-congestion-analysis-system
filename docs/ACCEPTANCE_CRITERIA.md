# 分阶段验收标准

## 1. 验收原则

- 验收以可复现证据为准，不以“页面能打开”或“模型能运行”代替质量验收。
- 所有数字必须能追溯到固定版本的数据或模型产物。
- 任一阶段未满足硬性条件，不进入下一阶段或合并 `main`。
- 范围变化先更新设计/规格并确认，不通过实现绕过基线。

## 2. M0：仓库与开发管理基线

- [ ] 本地仓库关联 `https://github.com/MUERJUN/traffic-congestion-analysis-system.git`。
- [ ] `main` 包含且保留 Phase 1/2 文档。
- [ ] `docs/DEVELOPMENT_PLAN.md` 明确双轨路线、依赖和分支策略。
- [ ] `docs/MODEL_TRACK_SPEC.md` 明确清洗、标签、特征、三模型、五指标与解释。
- [ ] `docs/DASHBOARD_TRACK_SPEC.md` 明确四页面和历史回放定位。
- [ ] 本文件定义后续各阶段验收条件。
- [ ] README包含项目介绍、当前进度、数据来源和开发路线。
- [ ] 原始数据和未来模型产物已被 `.gitignore` 保护。

## 3. M1：数据清洗与正式EDA

### 数据处理

- [ ] 原始 HDF5 哈希与来源登记一致且未被修改。
- [ ] 长表主键 `timestamp + sensor_id` 唯一，转换行数可解释。
- [ ] 207个传感器、时间范围和五分钟顺序保持一致。
- [ ] 0值、全网同步0、长连续0和大幅跳变均有显式质量标记。
- [ ] 系统性长缺口未使用双向或未来数据插值。
- [ ] 输出清洗前后记录数量、排除原因和覆盖率对照。

### EDA

- [ ] 完成时间规律、道路排行、速度趋势、拥堵前兆四项分析。
- [ ] 主要分析与保留0敏感性对照均有结果。
- [ ] 每张关键图包含单位、样本量/覆盖率和筛选口径。
- [ ] 每项至少有一条包含定量证据、业务意义和限制的结论。
- [ ] 不把统计关联表述为因果结论。

## 4. M2：标签、特征与时间切分

- [ ] 标签为未来30分钟持续拥堵，规则与设计基线一致。
- [ ] 自由流速度只使用训练期有效数据计算。
- [ ] 0.50/0.60/0.70速度比阈值敏感性分析完成。
- [ ] 所有滞后、滚动和历史同期特征只使用预测时点及之前的数据。
- [ ] 按时间70%/10%/20%划分，未随机打乱。
- [ ] 切分边界按历史窗口和未来窗口隔离。
- [ ] 特征字典记录来源、单位、窗口和缺失处理。
- [ ] 自动防泄漏检查通过并有报告。
- [ ] 训练、验证、测试的时间范围、样本数和标签比例均已登记。

## 5. M3：模型训练、比较与解释

### 必选模型

- [ ] Logistic Regression已完成。
- [ ] Random Forest已完成。
- [ ] XGBoost已完成。
- [ ] 三模型使用相同测试集、标签和核心特征口径。

### 指标与选择

- [ ] 每个模型输出Accuracy、Precision、Recall、F1-score、ROC-AUC。
- [ ] Precision、Recall和F1明确针对拥堵正类。
- [ ] 输出类别分布、混淆矩阵和ROC曲线。
- [ ] 最终模型首先按拥堵类F1比较，其次Recall、ROC-AUC。
- [ ] 测试集未用于调参、预处理拟合或阈值反复选择。
- [ ] 输出高峰/非高峰、工作日/周末及传感器分组误差分析。

### 解释与报告

- [ ] 输出全局特征重要性及方向说明。
- [ ] 至少三个代表性预测输出3—5个局部影响因素。
- [ ] 解释包含实际特征值、贡献方向和通俗业务含义。
- [ ] 完成模型比较报告，说明入选与未入选原因。
- [ ] 报告明确数据年代、覆盖范围、相关性/因果边界和非实时定位。

- [ ] 主要交互在目标普通电脑上达到约3秒响应目标，例外有记录。

## 8. Pull Request最低要求

- 说明所属轨道、阶段、范围和不在范围内的事项。
- 列出修改文件、验证命令和验证结果。
- 声明是否影响数据口径、标签、产物模式或页面文案。
- 不夹带另一轨道无关重构。
- 合并前解决与对应规格冲突的事项，并更新必要文档。
