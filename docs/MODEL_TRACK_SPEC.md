# Track A：算法与数据分析规格

## 1. 分支与职责

- 计划分支：`feature/model-pipeline`
- 工作范围：原始数据 → 清洗 → 正式EDA → 拥堵标签 → 特征工程 → 模型训练 → 模型评价 → 模型解释。
- 本轮状态：仅形成规格，不实现代码、不训练模型。

## 2. 输入基线

- 核心速度文件：`data/raw/metr-la.h5`，34,272×207。
- 同源元数据：传感器编号、经纬度、道路距离、207×207邻接矩阵。
- 时间范围：2012-03-01 00:00 至 2012-06-27 23:55。
- 采样粒度：5分钟，时间轴无缺口。
- 质量事实：无NaN、无负值；0值占8.1094%，包含大量全网同步0。

完整依据见 `reports/data_audit`，实现不得绕过审计结论直接把0定义为拥堵。

## 3. 数据清洗要求

### 3.1 不可变原始层

- 原始 HDF5 只读，不覆盖、不原位填补。
- 清洗结果写入 `data/interim` 或 `data/processed`，并记录输入哈希、参数和输出统计。

### 3.2 0值处理

- 保留 `speed_raw`，另建 `speed_valid` 和质量标记。
- 全网207个传感器同时为0的时点标记为系统性缺测候选。
- 同时至少50%传感器为0的时点进入系统质量复核。
- 单传感器长连续0标记为点位缺测候选。
- 主要EDA和建模口径不得把系统性缺测0当作真实拥堵。
- 必须提供“0作为缺测”与“保留0”的敏感性对照，说明结论差异。
- 不对长缺口进行线性插值；任何短缺口填补只能使用过去数据或训练期统计。

### 3.3 长表与质量校验

- 逻辑主键：`timestamp + sensor_id`。
- 理论长表行数：7,094,304；任何减少必须可由质量规则解释。
- 校验时间顺序、唯一主键、传感器集合、速度范围、质量标记数量和位置匹配率。

## 4. EDA要求

必须完成：

1. 不同时间段速度/拥堵规律。
2. 易拥堵道路观测点排行。
3. 道路速度变化趋势。
4. 拥堵发生前60分钟的数据变化特点。

每项必须包含研究问题、口径、有效样本量、图表、定量发现、业务含义和限制。不能只输出图表或只描述“高峰影响大”。

## 5. 标签定义

### 5.1 预测单元

一个样本对应 `sensor_id + prediction_timestamp`，目标是判断未来30分钟是否发生持续拥堵。

### 5.2 自由流速度

- 每个传感器自由流速度默认取该传感器训练期有效速度的第85百分位数。
- 验证集和测试集只能复用训练期结果，禁止重新计算。

### 5.3 拥堵状态与目标

- 速度比 = 有效速度 / 训练期自由流速度。
- 单时点拥堵默认定义为速度比 < 0.60。
- 对0.50、0.60、0.70阈值进行敏感性分析。
- 未来6个五分钟观测中至少3个拥堵，定义为未来30分钟持续拥堵标签1；否则为0。
- 未来窗口不完整或质量不足的样本不生成监督标签，并记录排除原因。

## 6. 时间切分与防泄漏

- 按全局时间排序切分：训练70%、验证10%、测试20%。
- 不随机打乱，不使用普通随机K折。
- 所有填补统计、自由流速度、标准化、类别编码和历史同期统计只在训练期拟合。
- 滚动与滞后特征必须先按传感器排序，只允许向过去取值。
- 切分边界按最长历史窗口和未来标签窗口做隔离，避免窗口跨集合污染。
- 测试集只用于最终一次正式评价，不参与阈值选择和调参。

## 7. 特征范围

### 核心特征

- 时间：小时、星期、周末标记、周期正余弦编码。
- 当前状态：当前有效速度、速度比、与历史同期差异。
- 滞后：5、10、15、30、60分钟历史速度/速度比。
- 滚动：过去15、30、60分钟均值、最小值、标准差。
- 趋势：最近10/15/30分钟斜率、相邻差分、连续下降次数。
- 历史同期：由训练期构造的传感器×日期类型×时段统计。

### 可选特征

邻近传感器过去时刻状态只能作为独立扩展实验；必须使用同源邻接元数据、确保时间对齐并做有/无空间特征消融比较。

## 8. 必选模型

1. Logistic Regression：透明线性基线。
2. Random Forest：非线性树集成。
3. XGBoost：梯度提升树模型。

另设 Dummy 或历史同期规则作为最低基线，但不能替代三个必选模型。不引入LSTM、DCRNN、Transformer等深度学习模型。

## 9. 训练和调参

- 三模型使用相同样本、标签、时间切分和核心特征口径。
- 类别不平衡优先使用类别权重或 `scale_pos_weight`，不默认全局过采样。
- 调参只使用训练/验证阶段；记录参数、随机种子、依赖版本和运行时间。
- 预处理与模型组成统一流水线，避免训练/推理字段错位。

## 10. 评价与选择

每个模型在固定测试集输出：

- Accuracy
- Precision（拥堵类）
- Recall（拥堵类）
- F1-score（拥堵类）
- ROC-AUC

同时输出混淆矩阵、ROC曲线、类别比例；建议补充PR-AUC和校准曲线。

最终选择顺序：

1. 先排除过拟合、时间外推不稳定或概率明显失真的模型。
2. 优先比较拥堵类别 F1。
3. F1接近时依次比较 Recall、ROC-AUC。
4. 指标接近时优先解释稳定、部署简单的模型。

## 11. 模型解释

- 全局：Logistic系数、排列重要性、树模型/SHAP汇总。
- 局部：每次预测输出3—5个推动风险上升或下降的因素、实际值和通俗说明。
- 必须区分“模型判断依据/统计关联”与“交通因果原因”。
- 解释结果需在时间段和传感器分组上检查稳定性。

## 12. 输出产物契约

Track A 至少向 Track B 提供以下版本化产物：

| 产物 | 建议格式 | 必要内容 |
|---|---|---|
| EDA汇总 | Parquet/CSV + JSON | 趋势、时间规律、道路排行、覆盖率 |
| 模型比较 | JSON/CSV | 模型名、五项指标、阈值、数据版本 |
| 最终模型 | Joblib或模型原生格式 | 预处理与模型完整流水线 |
| 特征字典 | JSON/Markdown | 字段、方向、时间窗口、单位 |
| 全局重要性 | CSV/JSON | 特征、重要性、方向和版本 |
| 预测样例/接口结果 | JSON | 时点、传感器、概率、等级、主要因素 |
| 业务建议规则 | JSON/Markdown | 触发条件、建议、限制 |

每个产物必须包含或伴随：生成时间、数据哈希/版本、模型版本、标签口径和字段模式。模型二进制及大体积生成物默认不提交Git。

## 13. 模型比较报告

正式报告必须包含：数据与标签、时间切分、预处理、特征、三模型参数、五项指标表、混淆矩阵、ROC曲线、误差分析、最终模型选择理由、全局/局部解释、局限和复现信息。

具体验收条件见 `docs/ACCEPTANCE_CRITERIA.md`。
